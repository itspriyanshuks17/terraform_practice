What is Terraform?

Terraform is an open-source Infrastructure as Code (IaC) tool developed by HashiCorp. It lets you define, provision, and manage cloud infrastructure using declarative configuration files instead of manual clicks in a cloud console.

Declarative

You describe what you want, not how to get there. Terraform figures out the steps.

Cloud-agnostic

Supports AWS, Azure, GCP, Kubernetes, GitHub, and 3,000+ providers.

Idempotent

Running the same config multiple times produces the same result — no duplicate resources.

State-aware

Tracks the current state of your infra in a .tfstate file for accurate change detection.

Why Terraform over clicking in the AWS Console? — Version control your infra, collaborate with teams, reproduce environments, audit changes, and automate deployments in CI/CD pipelines.
02

IaC Concepts

ConceptDescription
IaCManaging infrastructure through machine-readable definition files instead of manual processes
DeclarativeSpecify desired end state; tool determines how to achieve it (Terraform, CloudFormation)
ImperativeSpecify each step explicitly (Ansible, Bash scripts)
Desired StateWhat you declare in .tf files — what infra should look like
Current StateWhat actually exists in the cloud right now (tracked by state file)
DriftWhen current state diverges from desired state (e.g., someone manually changed something)

Installation & Setup

Ubuntu / Debian
# Add HashiCorp GPG key and repo
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor | \
  sudo tee /usr/share/keyrings/hashicorp-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install terraform

# Verify
terraform version

# Or export as environment variables

export AWS_SECRET_ACCESS_KEY="your-secret"
export AWS_DEFAULT_REGION="ap-south-1"

HCL Syntax

HCL (HashiCorp Configuration Language) is the language Terraform configs are written in. It is human-readable, declarative, and supports comments, variables, and expressions.

Basic HCL Structure
# Block type "resource", type "aws_instance", name "web"
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"   # argument = value
  instance_type = "t2.micro"

  tags = {
    Name = "MyServer"
    Env  = "dev"
  }
}
ComponentDescriptionExample
BlockTop-level construct that defines configresource {}, provider {}, variable {}
ArgumentKey-value pair inside a blockinstance_type = "t2.micro"
ExpressionDynamic value (reference, function call)var.region, aws_instance.web.id
InterpolationEmbed expressions in strings"server-${var.env}"
CommentSingle or multi-line# inline or /* block */

Providers

Providers are plugins that translate Terraform configurations into API calls for a specific platform (AWS, Azure, GCP, etc.).

Configuring the AWS Provider
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"    # ~ means >= 5.0 and < 6.0
    }
  }
  required_version = ">= 1.3"
}

provider "aws" {
  region = "ap-south-1"    # Mumbai region
  # Credentials come from env vars or ~/.aws/credentials
}
terraform init downloads the provider plugin specified in the config. Always run it first in a new project.

Resources

Resources are the core building blocks — they represent cloud infrastructure objects (EC2 instances, S3 buckets, VPCs, etc.).

Common AWS Resources
# EC2 Instance
resource "aws_instance" "app" {
  ami           = "ami-0f5ee92e2d63afc18"
  instance_type = "t2.micro"
  key_name      = "my-key"
  tags = { Name = "app-server" }
}

# S3 Bucket
resource "aws_s3_bucket" "data" {
  bucket = "my-terraform-demo-bucket-2025"
  tags   = { Environment = "dev" }
}

# Security Group
resource "aws_security_group" "web" {
  name = "allow-http-ssh"
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
Reference another resource's attribute using resource_type.resource_name.attribute: e.g., aws_instance.app.id or aws_vpc.main.id. Terraform builds a dependency graph automatically.

Variables & Outputs

Input Variables (variables.tf)
variable "instance_type" {
  type        = string
  default     = "t2.micro"
  description = "EC2 instance type"
}

variable "environment" {
  type    = string
  # No default → user must provide a value
}

variable "allowed_ports" {
  type    = list(number)
  default = [22, 80, 443]
}

# Usage in main.tf
instance_type = var.instance_type
Passing Variable Values
# Method 1: terraform.tfvars file (auto-loaded)
instance_type = "t3.small"
environment   = "prod"

# Method 2: CLI flag
terraform apply -var="environment=staging"

# Method 3: Environment variable
export TF_VAR_environment=staging
Local Values & Outputs
# locals.tf — reusable computed values
locals {
  common_tags = {
    Project     = "MyApp"
    Environment = var.environment
    ManagedBy   = "Terraform"
  }
  name_prefix = "${var.environment}-${var.project}"
}

# outputs.tf — expose values after apply
output "instance_public_ip" {
  value       = aws_instance.app.public_ip
  description = "Public IP of the EC2 instance"
}

State Management

The state file (terraform.tfstate) is how Terraform tracks what infrastructure it has created. It maps your config to real-world resources.

terraform.tfstate

JSON file auto-generated after apply. Contains resource IDs, attributes, dependencies.

terraform.tfstate.backup

Previous state saved before each operation. Used for rollback reference.

Never manually edit the state file! Use terraform state commands instead. Corruption can cause Terraform to lose track of your infra.
State Commands
terraform state list              # list all resources in state
terraform state show aws_instance.app  # show resource details
terraform state rm aws_instance.app    # remove from state (doesn't destroy resource)
terraform state mv old.resource new.resource  # rename/move
terraform import aws_instance.app i-1234567  # import existing resource

Modules

Modules are reusable packages of Terraform configuration. Think of them as functions — encapsulate related resources, accept inputs, return outputs.

Module Structure
modules/
  ec2-instance/
    main.tf        # resource definitions
    variables.tf   # input variables
    outputs.tf     # exposed values
Calling a Module
module "web_server" {
  source        = "./modules/ec2-instance"    # local path
  instance_type = "t2.micro"
  ami_id        = "ami-0f5ee92e2d63afc18"
  environment   = "prod"
}

# Using a public Registry module
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
  name    = "my-vpc"
  cidr    = "10.0.0.0/16"
}

# Reference module outputs
subnet_id = module.vpc.private_subnets[0]

Data Sources

Data sources let you query existing infrastructure or external data to use in your config — without managing it through Terraform.

Fetching an Existing AMI
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]    # Canonical

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-22.04-amd64-server-*"]
  }
}

# Use it in a resource
resource "aws_instance" "web" {
  ami = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"
}

Workspaces

Workspaces let you manage multiple environments (dev, staging, prod) from the same configuration with separate state files.

Workspace Commands
terraform workspace list              # show all workspaces (* = current)
terraform workspace new staging       # create and switch to "staging"
terraform workspace select prod       # switch to "prod"
terraform workspace show              # show current workspace

# Use workspace name in config
resource "aws_instance" "app" {
  tags = {
    Environment = terraform.workspace
  }
}

Remote State (S3 + DynamoDB)

In teams, store state remotely so everyone works from the same state. DynamoDB provides locking to prevent concurrent modifications.

Backend Configuration
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "prod/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "terraform-lock-table"   # for state locking
    encrypt        = true
  }
}
Create the S3 Bucket & DynamoDB Table First
# S3 bucket for state
resource "aws_s3_bucket" "tf_state" {
  bucket = "my-terraform-state-bucket"
}
resource "aws_s3_bucket_versioning" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  versioning_configuration { status = "Enabled" }
}

# DynamoDB for locking
resource "aws_dynamodb_table" "tf_lock" {
  name         = "terraform-lock-table"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute {
    name = "LockID"
    type = "S"
  }
}

Lifecycle Rules

lifecycle Block Options
resource "aws_instance" "app" {
  ami           = "ami-12345"
  instance_type = "t2.micro"

  lifecycle {
    create_before_destroy = true   # create replacement before destroying old
    prevent_destroy       = true   # block accidental terraform destroy
    ignore_changes        = [tags] # ignore tag changes (manual edits won't trigger plan)
  }
}
OptionWhen to use
create_before_destroyZero-downtime replacements (immutable infra)
prevent_destroyProduction databases, state buckets — anything critical
ignore_changesWhen resources are modified outside Terraform (e.g., auto-scaling)

CLI Reference

1

terraform init

Downloads provider plugins, initializes backend, sets up working directory. Run first in any project.

2

terraform validate

Checks .tf files for syntax errors and configuration validity without accessing cloud APIs.

3

terraform fmt

Auto-formats all .tf files to standard HCL style. Use -recursive for nested directories.

4

terraform plan

Shows what will be created, changed, or destroyed. Output uses + (create), ~ (update), - (destroy). Save with -out=plan.tfplan.

5

terraform apply

Executes the plan, creates/updates real infrastructure. Use -auto-approve in CI/CD pipelines.

6

terraform destroy

Tears down all managed infrastructure. Irreversible — use with caution in prod. Use -target to destroy specific resources.

Other Useful Commands
terraform output                    # show output values
terraform output instance_public_ip # specific output
terraform show                      # human-readable state
terraform graph | dot -Tsvg > g.svg # visualize dependency graph
terraform force-unlock LOCK_ID      # manually release stuck state lock
terraform taint aws_instance.app    # mark resource for recreation on next apply

Project 1 — EC2 + S3 Deployment

Project 01 / Beginner

Deploy an EC2 instance with an S3 bucket

Provision a basic AWS infrastructure with a web server EC2 instance, security group, and an S3 bucket for static file storage. Covers provider setup, resource creation, and outputs.

EC2 S3 Security Group Outputs
main.tf
provider "aws" { region = "ap-south-1" }

resource "aws_instance" "web" {
  ami                    = "ami-0f5ee92e2d63afc18"
  instance_type          = var.instance_type
  vpc_security_group_ids = [aws_security_group.web_sg.id]

  user_data = <<-EOF
    #!/bin/bash
    apt-get update -y
    apt-get install -y nginx
    systemctl start nginx
    systemctl enable nginx
  EOF

  tags = { Name = "terraform-web-server" }
}

resource "aws_s3_bucket" "assets" {
  bucket = "terraform-demo-assets-${random_id.suffix.hex}"
}

resource "random_id" "suffix" {
  byte_length = 4
}
Run order: terraform init → terraform plan → terraform apply. After apply, terraform output to see the EC2 public IP.

Project 2 — Custom VPC + Full Infra

Project 02 / Intermediate

Production-ready VPC with public & private subnets

Build a complete networking layer: VPC, Internet Gateway, Route Tables, public subnets, EC2 in public subnet. Introduces resource dependencies and CIDR planning.

VPC Subnets IGW Route Tables EC2
VPC + Networking (main.tf)
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  tags = { Name = "terraform-vpc" }
}

resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "terraform-igw" }
}

resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "ap-south-1a"
  map_public_ip_on_launch = true
  tags = { Name = "public-subnet" }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
}

resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id

Project 03 / Advanced

Multi-environment infra using custom modules

Create reusable modules for EC2, VPC, and Security Groups. Deploy multiple environments (dev, staging, prod) from the same codebase by passing different variable files. Demonstrates DRY Terraform architecture.

Best Practices

  • Always run terraform plan first — never apply blindly. Review what will change.
  • Use remote state in teams — S3 + DynamoDB locking prevents state corruption.
  • Never commit .tfstate files — add *.tfstate and *.tfstate.backup to .gitignore.
  • Never hardcode credentials — use IAM roles, env vars, or AWS Secrets Manager.
  • Pin provider versions — use version = "~> 5.0" to avoid breaking changes.
  • Modularize your code — one module per logical unit (networking, compute, database).
  • Tag all resources — use locals for consistent tagging across all resources.
  • Use prevent_destroy on production databases, S3 state buckets, and other critical resources.
  • Separate state per environment — dev, staging, and prod should each have their own state.
  • Use terraform fmt and terraform validate in CI before every merge.
